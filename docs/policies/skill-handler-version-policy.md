# Skill Handler Version Policy

## Purpose

Executable skill handlers are production runtime code. They must stay aligned
with module metadata, runtime profile version pins, static coverage evidence,
and live release evidence. This policy defines how handler versions are added,
deprecated, rolled back, and certified.

The machine-readable policy is:

```text
policies/runtime/skill-handler-version-policy.json
```

Its schema is:

```text
schemas/skill-handler-version-policy.schema.json
```

## Binding Rule

The runtime binds a selected skill only by exact `skill name + version`.
The selected skill name must appear in `profile.skills`, and the selected
version must match `profile.module_versions[skill]`. A missing handler or
version mismatch is a policy denial before tool execution.

Runtime profiles are immutable. An upgrade or rollback must create a new
profile through the Composer with a different version pin; the runtime must not
edit the active profile in place.

## Upgrade Rule

Add a new handler version side by side with the previous version:

1. Add or update the module metadata for the new version.
2. Register the new executable handler with the exact same skill name and new
   semantic version.
3. Keep the previous handler registered until static coverage, live handler
   binding evidence, and release evidence pass for the new version.
4. Mark old versions as `deprecated` only after no default production profile
   selects them.

The coverage manifest records each handler lifecycle status. Current values are
`active` and `deprecated`.

## Deprecation Rule

Deprecated handlers remain executable for rollback while any active release,
runtime profile, or rollback plan may still reference them. Removing a
deprecated handler is allowed only when:

- no committed module fixture or production profile pins that version,
- no pending release evidence references that version,
- rollback no longer requires the handler, and
- the coverage manifest and tests are updated in the same change.

## Rollback Rule

Rollback must restore or select a previous registered handler version through a
reviewed repository change and a newly composed runtime profile. Rollback must
preserve fail-closed behavior:

- do not mutate the active profile,
- do not route by task type alone,
- do not introduce a free-form handler entrypoint,
- rerun static coverage validation,
- rerun live handler binding evidence before certification.

## Release Evidence

Production certification must include:

- static coverage from `examples/runtime/skill-handler-coverage.json`,
- live handler binding evidence from the `live-runtime-handler-binding-evidence`
  artifact,
- version policy tests from `tests/test_skill_handler_version_policy.py`, and
- release evidence generated by
  `scripts/release/build_production_readiness_evidence.py`.
